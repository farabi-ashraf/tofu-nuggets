> Standing instructions from the owner for the next version. A future Claude session

## Field report that triggered this version

First external install (Win 10 Pro 19045, x64): hotkey + settings dead. Resolved by the
tester manually installing the **WebView2 Evergreen Standalone Installer** and
reinstalling Tofu Nuggets. Conclusion: our NSIS default (`downloadBootstrapper`) needs
network at install time and can fail silently, leaving the app tray-alive but unable to
build any webview window.

## A. Fixes

### A1. WebView2 install reliability (do first, one config change) — ✅ DONE (2026-07-20)

> Implemented in `0f39e3f`, verified 2026-07-20: forced `overlay::create` failure →
> native MessageBox shown (title/text/Yes-No correct), failure logged to tofu.log,
> exit code on dismiss; installer rebuilt with embedded bootstrapper
> (2.9 MB → 4.66 MB, build step downloads fwlink 2124703 to embed).

- Set `bundle.windows.webviewInstallMode` to `{ "type": "embedBootstrapper" }` in
  `tauri.conf.json` (installer grows ~2 MB; works offline-ish, retries properly).
  Consider `offlineInstaller` (~130 MB) only if testers keep hitting it.
- Also: detect the failure at runtime — if `overlay::create` fails at startup, show a
  native MessageBox pointing at the WebView2 download page instead of dying silently.
  (Win32 `MessageBoxW`, no webview needed by definition.)

### A2. Badge dots: persist, never flicker, never overlap other windows

> **Spike verdict (2026-07-20, `spikes/badge-reparent`): reparent-under-WorkerW is
> NOT viable on Win11 26200.** The desktop chain refuses to composite foreign
> windows: layered (ULW/colorkey/alpha), WS_CHILD, WS_EX_TRANSPARENT, and
> SetWindowRgn-shaped windows never render; plain rect windows render only
> erratically (size/position dependent). Full findings table in the spike README.
> **Plan B (approved + implemented 2026-07-20, `badges.rs` on `wip-a2-badges`):**
> kept the TOPMOST layer; (1) per-dot occlusion — a dot is skipped while any
> visible/non-cloaked/non-minimized top-level window overlaps its pixels, so
> dots never bleed over windows; (2) SetWinEventHook (FOREGROUND +
> LOCATIONCHANGE, 80 ms coalescing) replaces the 2 s foreground show/hide gate —
> dots persist whenever the desktop is visible; (3) 2 s timer keeps icon/sidecar
> state, skipping the UIA walk while a fullscreen window covers everything;
> repaint only when the visible-dot set changes. `desktop_is_foreground()`
> removed (orphaned).
> **Verified**: dot renders on badged icon; dots under an overlapping app window
> stay retracted (screenshots); idle 0.05% CPU / 7.5 MB private (debug build).
> **Owner still to eyeball**: alt-tab flicker feel, settings badges-toggle,
> drag-a-window-over-a-dot latency (~100 ms design).

Owner: dots should simply persist while enabled in settings — no appear/disappear on
alt-tab (current 2 s foreground-gated refresh makes them lag in and out; distracting).
And they must never draw over other windows (today the layer is TOPMOST, so with the
desktop focused, dots bleed over overlapping inactive windows).

Both symptoms have one root cause: the badge layer floats above everything, so we hide
it whenever the desktop is not foreground. **Planned fix: put the layer into the
desktop's own z-order instead** — reparent the badge window under `WorkerW`/`Progman`
(the wallpaper/icon window chain, same trick desktop-widget apps use), drop
`WS_EX_TOPMOST`, drop the `desktop_is_foreground()` show/hide gating entirely.
Result: dots sit just above the icons, below every application window, and persist.

- Keep the 2 s refresh only for icon-position/sidecar changes (cheap; consider
  event-driven later).
- Handle Explorer restart (WorkerW handle dies) — re-find + re-parent, same place we
  re-apply infotip suppression.
- Risk: WorkerW variants across wallpaper slideshow modes and Win 10 vs 11 — needs the
  same careful spike treatment as M0.
- Verify: alt-tab produces zero dot flicker; a window overlapping a badged icon covers
  the dot; dots visible whenever desktop is; settings toggle still works.

### A3. Hover panel: Edit and Close buttons

- Panel header gets two small icon buttons: ✎ Edit (invokes `edit_nugget` for the
  current path, hides panel) and ✕ Close (hides panel immediately instead of waiting
  for mouse-leave).
- Panel is non-focusable but receives clicks (todo-checkbox precedent, see
  ARCHITECTURE.md). Close needs either a tiny `hide_overlay` command or the
  `core:window:allow-hide` permission — command preferred (keeps capability surface
  small).
- Verify via CDP: click ✎ → editor opens for the right path; click ✕ → panel hides.

### A4. "Logitech G HUB" desktop icon can't take a note — investigate

Hypothesis: display name ≠ filename (installer-created shortcut with localized or
renamed display name), so name-based path resolution in `desktop.rs` fails and the icon
is treated as virtual. The log will say which branch fired
(`no desktop icon` / `virtual icon` / resolve failure) — ask the owner for
`%APPDATA%\com.tofunuggets.app\tofu.log` right after reproducing.
Fix direction if hypothesis holds: resolve the filesystem path from the shell item
behind the ListView entry (IShellItem / desktop folder view) instead of matching the
display name against directory entries.

## B. Infrastructure

### B1. Updates via GitHub ("Check for updates" in tray)

Yes, GitHub is the right backbone; this is the standard Tauri path:

> **Progress (2026-07-19):** step 1 DONE — repo published **private** at
> `https://github.com/farabi-ashraf/tofu-nuggets`, default branch renamed `master` → `main`,
> full history pushed. Added `README.md`; untracked `.claude/settings.local.json`
> (now gitignored). Steps 2–6 below still pending.

1. **Repo**: ✅ created and pushed (see Progress note). **Still to do before the updater
   works:** releases must be public for the updater to fetch without tokens — either flip
   the repo public, or keep it private but publish public releases. The repo is private
   for now; decide at first-release time.
2. **Signing**: `npx tauri signer generate` → keypair. Public key goes in
   `tauri.conf.json` (`plugins.updater.pubkey`); private key + password become GitHub
   Actions secrets (`TAURI_SIGNING_PRIVATE_KEY`, `..._PASSWORD`). Never commit it.
3. **Plugin**: add `tauri-plugin-updater` + `tauri-plugin-dialog` confirm flow. Endpoint:
   `https://github.com/<owner>/tofu-nuggets/releases/latest/download/latest.json`.
4. **Tray**: add "Check for updates…" item → `updater.check()` → dialog with version +
   notes → download & install → restart. Also log to tofu.log.
5. **CI**: GitHub Actions with `tauri-apps/tauri-action` — on pushing a tag `v*`, it
   builds the installer, signs it, and publishes a GitHub Release **with `latest.json`
   generated automatically**. Releasing 0.1.2 then = bump version in tauri.conf +
   `git tag v0.1.2 && git push --tags`. Testers click Check for updates.
6. NSIS + updater: updater-compatible artifacts require `createUpdaterArtifacts: true`
   in bundle config.

### B2. Multi-machine / multi-OS strategy

**No separate branches — ever.** One codebase, one `main`; platform differences live
in the code behind `#[cfg(...)]`/traits, and per-platform installers come from a CI
build matrix building the *same commit*. "Change one thing → update all versions" is
then automatic: one commit, one tag, CI produces every installer. Branch-per-platform
means hand-porting every fix N times — the maintenance failure mode to avoid.

- **Now**: Windows x64 only (what we ship).
- **Cheap when wanted**: Windows arm64 — add `aarch64-pc-windows-msvc` target to the CI
  matrix; `windows` crate + WebView2 support it. x86 (32-bit): skip, dead market.
- **macOS/Linux (post-MVP per FEASIBILITY)**: frontend, storage, index, watcher, editor
  all carry over. The Windows-only organs are `desktop.rs` (UIA hover detection),
  `badges.rs` (layered window), overlay positioning, and infotip suppression. Prep step
  when the time comes: extract a `DesktopIcons` trait (icon_at / list_icons /
  selected_icon / suppress_tips) with the Win32 impl behind `#[cfg(windows)]`, then
  write AX-API (macOS) / per-DE (Linux) impls. Do NOT abstract preemptively — wait
  until the second platform is actually started.

### B3. Uninstall & data lifecycle

Three data locations, three different owners:

| Data | Where | On update/reinstall | On uninstall |
|---|---|---|---|
| Notes (sidecars) | `.nuggets` folders next to the user's files | untouched (by design — survives everything) | untouched — uninstaller must never sweep the disk |
| Index + settings + log | `%APPDATA%\com.tofunuggets.app` | untouched → settings/hotkey survive updates | NSIS "delete app data" checkbox removes this only |
| Program | `%LOCALAPPDATA%\Tofu Nuggets` | replaced | removed |

The tester's observation (".nuggets still there after checking delete-data") is the
uninstaller working as designed: that checkbox covers app-data only, and sidecars are
user documents scattered beside user files — an uninstaller crawling the disk to delete
them would be dangerous and slow. **Planned fix — make the intent explicit in-app:**

- Settings gains a danger-zone section: **"Delete all notes…"** — two-step confirm,
  walks the index (plus a desktop-dirs scan for strays), deletes every sidecar and
  empty `.nuggets` dir, then clears the index. Message in the uninstaller flow and the
  main-window empty state: "to remove your notes too, use Settings → Delete all notes
  before uninstalling."
- Update path already ports everything (sidecars re-scanned, settings preserved) —
  keep it that way; never key data to the install dir.
